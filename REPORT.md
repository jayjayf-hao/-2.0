# snakeII 设计与实验报告

## 1. 核心目标完成情况
- Game Loop 采用标准三段结构：`processInput -> update -> render`，位于 `core/GameLoop.cpp`，主函数仅负责启动。
- 输入系统实现“同帧多输入 + 最新合法输入覆盖策略”，非法 180 度反向输入被过滤。
- 使用 `isOpposite(dir1, dir2)` 统一约束方向合法性，避免反向自撞。
- 时间系统采用固定时间步长（Fixed Timestep），逻辑更新由 `tickInterval` 控制，非 `Sleep(100)`。
- FSM 完整实现：`MENU -> RUNNING -> PAUSED -> GAME_OVER -> EXIT`，输入行为按状态绑定。
- 输入采集、状态更新、渲染完全解耦：输入可高频发生，逻辑固定节奏更新。
- 鲁棒性已覆盖：非法键过滤、快速连按稳定、坏档检测并给出错误信息。

## 2. 为什么不用队列
- 队列会保留历史输入，容易出现“玩家已经改主意，但系统仍在执行旧指令”的滞后感。
- 贪吃蛇控制目标是“当前意图优先”，不是“历史命令必执行”。
- 在高速移动下，队列还会造成方向执行延迟，手感变钝。

## 3. 为什么不能只取第一个输入
- 同一帧用户可能完成“修正输入”（例如先误按再立刻纠正）。
- 只取第一个输入会吞掉用户最后意图，导致反应不符合直觉。
- 最新合法输入覆盖策略可以在保持安全（过滤反向）的同时获得更高响应性。

## 4. 时间系统说明
- 逻辑更新时间条件：当累计时间 `accumulator >= tickInterval` 时执行一次 `update`。
- 采用三档速度：LOW/MEDIUM/HIGH。
- 支持动态加速：分数提升后自动切换更高速度档。

## 5. 对比实验

### 实验1：直接输入（每次按键立即改方向，无同帧筛选）
- 现象：快速按键时容易出现误触发。
- 风险：若无反向过滤，可能瞬间 180 度反转自撞。

### 实验2：输入队列（FIFO 顺序执行）
- 现象：高频输入时存在明显延迟。
- 问题：可能执行“历史输入”，与玩家当前意图不一致。

### 实验3：最新合法输入覆盖（最终方案）
- 现象：在高频输入下更流畅，响应更接近玩家直觉。
- 安全性：非法反向输入在采集阶段即被过滤。
- 结论：在本项目中，该策略在响应性与稳定性之间取得最佳平衡。

## 6. 鲁棒性策略
- 非法输入：未映射按键直接忽略并计数。
- 连续输入压力：每帧读取完输入缓冲，再一次性给出最新合法方向。
- 存档损坏：逐字段校验范围和格式，失败时返回具体错误原因。
- 更新防爆：单帧最多处理有限次逻辑更新，避免积压导致卡顿螺旋。
